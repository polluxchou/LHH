# On-demand Brief Persistence

**Date:** 2026-06-17
**Status:** Implemented (retroactively documented). Commits `6597ef9` (initial) and the regenerate/verification follow-ups in `lib/account/content-mutations.ts` (`persistGeneratedBrief`).

## Problem

Clicking「生成简报」(generate brief) on a candidate signal only updated the
in-memory workbench store (`setContentState`). On refresh, `SpaceProvider`
rebuilds a space's content from the DB via `getSpaceContent`, so any brief that
was never written to the DB vanished. By contrast, the daily-search path
(`runSearchForObject` → ingest writer) writes to the DB and refreshes, so its
results persisted. Briefs needed the same.

## Decision

Persist an on-demand brief to the DB right after the optimistic in-memory commit,
mirroring the ingest writer's brief+score write. No schema change — the
`editorial_briefs` and `content_value_scores` tables already existed and were
already read by `getSpaceContent`. The only gap was the missing **write** on the
on-demand path.

## Data model (existing tables, no migration)

`persistGeneratedBrief(input)` (service-role, membership-checked via the tracking
object's `space_id`):

1. **Overwrite-safe**: delete any existing brief for `candidate_signal_id` in this
   space first (FK `on delete cascade` clears its `content_value_scores`). First
   generation matches nothing (no-op); regenerate replaces the old brief. This is
   why the same action also backs the「重新生成」path.
2. Insert `editorial_briefs` (brief_title, tagline, fact_bullets, fact_summary,
   source_summary, map_context, why_it_matters, possible_angles, open_questions,
   risk_notes, `verification`, status, space_id). DB generates the `uuid` id;
   `location_anchor_ids` left to its `'{}'` default (in-memory anchor ids are not
   DB uuids).
3. Insert `content_value_scores` (the 6 sub-scores + overall_recommendation +
   scoring_notes + space_id). `compositeScore` is not stored — derived at read
   time as `round(confidence × 100)`.

The brief's `candidate_signal_id` references an existing DB signal (candidate
signals are always DB-backed), so the FK holds.

## Flow

`generateBrief` / `regenerateBrief` (workflow-provider): compute the brief +
score in memory (instant feedback) → call `persistGeneratedBrief` → on failure,
log a localized `briefNotPersisted` warning (the in-memory brief still shows for
the session). No `router.refresh()` — the in-memory state already matches what the
DB now holds; a later refresh reloads the same brief from the DB.

## Verification (production)

Confirmed by querying `editorial_briefs` directly: an on-demand brief
(「飞沃科技…」, 2026-06-17, microsecond `created_at`) carried
`scoring_notes = "Generated by the local workflow from deterministic fixture
data."` — the fixed string only `createDefaultScore` (the on-demand path) emits,
proving the button's output reached the DB. (Search-path briefs carry AI-authored
scoring notes instead.)

## Rollout & rollback

- **Rollout:** code-only, no migration — normal `git push` / deploy.
- **Rollback:** revert the `persistGeneratedBrief` call in `generateBrief`/
  `regenerateBrief` and the action itself. No DB state to undo (the action only
  inserts rows the app already reads; orphaned rows, if any, are valid briefs).
- **Failure mode:** the action returns `{ok:false, reason}` on any DB error; the
  caller logs a warning and keeps the in-memory brief. Never throws into the UI.
